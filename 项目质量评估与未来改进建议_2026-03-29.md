# 群约小助手｜项目质量评估与未来改进建议

日期：2026-03-29  
评估范围：当前仓库主干版本（Flask + SQLite + 原生前端）

---

## 1. 结论摘要

当前项目已经从“功能能跑”的个人原型，进入到“可持续维护的小型上线项目”阶段。

核心判断：

- **产品可用性**：已达可交付水平，主流程完整，界面可读性和移动端体验明显优于早期版本。
- **工程质量**：中等偏上，具备基础测试、部署入口、健康检查、安全扫描脚本，但还未形成完整的工程化闭环。
- **上线稳定性**：适合低到中等规模使用；当并发、历史数据量、协作人数继续增长时，现有架构会先在数据层和可观测性层面暴露瓶颈。
- **未来方向**：P0 底座已补齐，下一阶段应转向更细的回归测试、数据模型约束和长期维护成本控制。

这部分质量明显高于常见的“能用但粗糙”的个人项目。

### 3) 后端已经开始体现工程约束

当前后端不是单纯“接收什么存什么”，而是已经开始做：

这说明项目开始从“脚本式服务”向“可维护服务”过渡。

### 4) 安全意识高于同阶段项目平均值

这类项目里，很多仓库在这个阶段还没有任何安全收口动作；当前项目在这方面已经建立了基础习惯。

### 5) 文档基础足够

目前仓库里已经有：

- README

文档虽然仍偏人工维护，但已经具备连续演进的上下文，不是“代码之外一片空白”的状态。

## 2.2 当前主要问题

### 1) 前端已完成首轮模块拆分，但模块边界还需要继续收紧

当前前端已经拆为入口、状态、API、历史、渲染、总结等模块，这一步已经把“单文件主逻辑”问题降下来了。

这在当前规模还能接受，但后续一旦继续增加：

- 更多筛选或视图能力
- 更复杂的 AI 总结展示
- 更强的表格交互
- 多语言、权限、复制/导出等能力

当前主要风险不再是“一个文件太大”，而是：

- 渲染模块仍然承担较多 DOM 拼接职责
- 交互逻辑和状态更新仍有继续细分空间
- 前端还没有模块级测试约束

结论：  
**P0 已完成，下一步应从“拆文件”转向“稳边界”。**

### 2) 测试覆盖仍明显不足

当前测试主要集中在 API 冒烟层，优点是已经覆盖最小主链路；不足也很明显：

- 没有前端交互测试
- 没有异常路径测试的系统覆盖
- 没有部署后端到端回归测试
- 没有 AI 总结输出结构稳定性测试
- 没有针对历史记录、轮询同步、视图切换的回归测试

结论：  
**当前测试足以防止“服务完全坏掉”，但不足以防止“局部体验回退”。**

### 3) 可观测性基础已建立，但还没有形成完整监控闭环

现在项目已经具备：

- 请求级结构化日志
- 请求 ID 回传
- 统一 API 错误结构
- 关键业务事件日志（创建、加入、保存、总结）

当前仍缺的部分是：

- 聚合后的日志检索视角
- 更明确的错误分级统计
- 慢请求监控与阈值告警

这会导致一个典型问题：  
**单次排障已经可做，但长期线上观测还不够系统。**

### 4) SQLite 方案适合当前阶段，但扩展性有限

SQLite 对这个项目当前阶段是合理选择，原因很直接：

- 部署成本低
- 运维简单
- 单人项目负担小
- 本地开发体验好

但它的上限也很明确：

- Railway 场景下需要显式处理持久化卷
- 写入并发提升时容易先碰到数据层瓶颈
- 后续若要统计分析、审计、复杂查询，能力会受限

结论：  
**短期不建议过早迁移数据库，但要把迁移条件提前定义清楚。**

### 5) 安全扫描和 CI 已落地，但规则能力仍偏轻

`security_guard.py` 已经进入仓库级门禁，但目前仍存在限制：

- 规则库较少
- 依赖正则，误报/漏报都可能存在
- 没有对 `.env`、日志产物、构建文件做更系统的策略区分

结论：  
**当前已具备最小自动化门禁，但还不是高强度安全治理。**

### 6) 运维文档已补恢复 SOP，但还缺实际演练记录

当前文档已经覆盖：

- 数据库怎么备份
- 数据库怎么恢复
- 发布前的数据库保护动作

仍未完全闭环的部分：

- 出现误删或坏数据时的实战演练记录
- 线上 AI 服务异常时的处置顺序
- 发布失败后的回滚记录模板

结论：  
**当前文档已可支撑基础运维，但还没经过足够演练。**

## 3.2 P1：应尽快安排，但不必抢在最前面的改进

### P1-1 扩展测试到 UI 关键回归

重点不是一开始就做很多，而是抓最容易回退的点：

- 创建者自动加入
- 历史记录恢复
- 人员/时间双布局切换
- AI 总结弹层渲染
- 备注保存与再次进入读取

建议方向：

- 若继续保持轻量，可先做更细的后端/工具层测试
- 若准备长期维护前端，建议引入浏览器级 E2E 测试

### P1-2 统一配置层

当前配置项还比较少，但可以提前收口：

- `DB_PATH`
- `DEEPSEEK_API_KEY`
- AI 超时
- worker 数
- 环境标识

建议形成单独配置模块，避免参数散落。

### P1-3 总结数据模型和存储策略

当前 session 数据直接 JSON 存 SQLite，适合当前阶段，但未来会碰到几个问题：

- 字段演化缺少迁移机制
- 审计和统计查询困难
- 局部更新粒度粗

建议短期先定义 schema 约束，长期再评估是否拆表。

## 3.3 P2：中期可选项，不宜现在就投入过多

### P2-1 从 SQLite 迁移到 PostgreSQL

这个动作只有在以下条件出现时才值得做：

- 明显的并发/持久化压力
- 需要更复杂的查询统计
- 需要更稳定的线上数据治理
- 准备做多环境部署和正式运维

在此之前直接迁移，收益不一定覆盖复杂度成本。

### P2-2 引入前端框架

如果未来功能继续明显膨胀，React/Vue 确实会降低状态和视图复杂度；但当前阶段直接迁移，风险也很清楚：

- 重写成本高
- 回归面大
- 项目复杂度会提前上升

判断标准很简单：  
**如果未来两三个版本内，前端交互复杂度继续显著增加，再迁移；否则继续在原生架构下做模块化拆分。**

### P2-3 用户体系 / 权限体系

现阶段项目定位是轻量协作工具，不是后台型系统。  
如果没有明确业务需求，过早引入登录、权限、组织结构，会让项目失去原本的轻量优势。

## 3.4 用户体验层面的下一步建议

下面这些不属于“底层工程化”，但会直接影响用户留存和完成率，优先级值得单独看待。

### UX-1 创建流程再缩短一步

当前创建页已经可用，但还可以进一步降低首屏思考成本：

- 提供常用模板，例如“工作日晚间”“周末全天”“本周 3 天”
- 自动给出默认活动名占位示例，而不是只靠用户自己想
- 在日期和时间段变化时，实时显示“共生成多少天、多少个时段”

价值评估：

- 能降低首次使用门槛
- 对移动端尤其有效

优先级：高  
实施成本：低到中

### UX-2 结果页增加“无需 AI 也能一眼看懂”的摘要层

现在有表格和 AI 总结，但中间还缺一层非常直接的产品摘要：

- 当前最佳时段 Top 3
- 已填写 / 未填写人数
- 冲突人数最多的时段
- 适合立即拍板的时段标记

价值评估：

- 不依赖 AI 也能快速决策
- 用户在群里截图传播时信息更直接

优先级：高  
实施成本：中

### UX-3 对“未完成填写”的人做更明显的提示

当前用户能看到参与者，但缺少更强的完成度反馈。建议增加：

- 已填写人数进度条
- 未填写人名单提示
- 若自己未填写，主操作区固定提示“继续填写”

价值评估：

- 这会直接提高表格完成率
- 比单纯展示名单更接近真实协作场景

优先级：高  
实施成本：低

### UX-4 分享链路继续压缩

当前复制链接已经有了，但还可以更顺手：

- 分享弹层里直接展示二维码或短链接
- 自动附带一段可复制文案
- 对发起人提供“提醒未填写成员”的快捷文案

价值评估：

- 对微信群、临时活动组织很有用
- 能显著减少“建好了但没人填”的问题

优先级：中到高  
实施成本：中

### UX-5 表格浏览增加更强的筛选视角

当前双布局已经很好，但还可以补两个高价值筛选：

- 只看最佳时段
- 只看我和某几个人的交集

价值评估：

- 当参与人数增多时，表格复杂度会快速上升
- 这些筛选比继续堆颜色更有价值

优先级：中  
实施成本：中

### UX-6 加强加载、失败和空态文案

当前功能层面已经完整，但体验上还可以更“产品化”：

- AI 总结生成中增加更明确的阶段提示
- 网络失败时显示可执行的重试文案
- 历史为空、无人填写、无人重合等空态更明确

价值评估：

- 这类文案不显眼，但会显著降低用户疑惑

优先级：中  
实施成本：低

### UX-7 补基础可访问性

当前已经有视觉提升，但还可以补：

- 键盘可操作性
- 更明确的焦点态
- 颜色之外的状态辅助
- 对屏幕阅读器更友好的结构说明

价值评估：

- 对广义可用性有帮助
- 也是产品成熟度的重要标志

优先级：中  
实施成本：中

### UX 层面的建议排序

如果只做一轮用户体验改进，建议顺序如下：

1. 创建流程模板化与时段数量提示
2. 结果页增加非 AI 的摘要层
3. 已填写 / 未填写进度提示
4. 分享链路压缩
5. 表格筛选视角

## 5. 最终判断

这个项目当前最大的优点，不是“功能很多”，而是**已经形成了产品感、工程底座和继续演进的基础**。

当前最大的风险，也不是“哪里完全不可用”，而是：

- 前端逻辑集中度偏高
- 测试和日志还不足以支撑更复杂迭代
- 运维恢复能力没有形成闭环

P0 已完成，因此最合理的下一步不是回头再做底座，而是开始补更细粒度的质量保证。

建议原则：

1. 先补更细的测试和配置治理，再继续扩功能。
2. 不急于上更重的技术栈，先把当前模块化后的原生架构用到上限。
3. 只在真实需求出现时再引入数据库迁移、前端框架和权限体系。

按这个路径推进，项目可以比较稳地从个人小项目进入长期可维护状态。
